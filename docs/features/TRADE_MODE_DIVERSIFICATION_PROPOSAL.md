# 거래 방식 다각화 기획 제안서

## 1. 제안 요약

현재 DAMARA의 공동구매는 기획상 다음 두 흐름으로 나눌 수 있다.

```text
A 타입: 선모집형
B 타입: 후모집형
```

여기에 프론트엔드 부담이 크지 않으면서도 공동구매 서비스의 재미를 살릴 수 있는 1차 확장으로
**선모집형 + 모이면 싸지는 공구**를 제안한다.

핵심 문구는 다음과 같다.

```text
2명만 더 모이면 가격이 내려가요.
```

## 2. 현재 방식 분석

현재 백엔드에는 `groupBuyType` 필드가 있다.

```text
groupBuyType
```

다만 현재 구현에서는 이 값이 enum이나 정책 로직으로 강하게 고정되어 있지 않고,
프론트에서 넘기는 문자열을 저장하고 반환하는 수준에 가깝다.

즉 현재 상태는 다음과 같다.

```text
기획상:
A/B = 선모집형 / 후모집형

백엔드 구현상:
groupBuyType 문자열 필드는 있으나,
선모집/후모집에 따른 별도 거래 정책은 아직 약함
```

현재 공동구매의 기본 흐름은 다음에 가깝다.

```text
작성자가 게시글 등록
참여자가 참여
최소 인원과 마감 시간 존재
수령 장소/시간 안내
참여자 상태 관리
거래 완료 또는 취소
```

이 구조는 안정적이지만, 사용자가 친구를 데려오거나 빠르게 참여해야 할 이유는 약하다.

## 3. 제안 기능

### 모이면 싸지는 공구

선모집형 공동구매에서 목표 인원을 달성하면 가격이 내려가는 방식이다.

예시:

```text
현재 가격: 5,000원
목표 인원: 5명
목표 달성 가격: 4,500원
현재 참여 인원: 3명
안내 문구: 2명만 더 모이면 4,500원
```

사용자는 단순히 참여만 하는 것이 아니라, 다른 사용자를 데려올 이유를 갖게 된다.

```text
내가 참여하면 목표에 가까워짐
친구를 데려오면 모두의 가격이 내려감
마감 전에 빨리 모을 동기가 생김
```

## 4. 프론트엔드 부담을 줄이는 MVP 방식

처음부터 다단계 가격표를 만들지 않는다.

1차 MVP는 한 단계만 제공한다.

```text
기본 가격
목표 인원
목표 달성 가격
```

프론트엔드가 복잡한 계산을 하지 않도록 백엔드가 계산 결과를 내려준다.

응답 예시:

```json
{
  "groupBuyType": "pre_recruit",
  "groupBuyMode": "price_unlock",
  "price": 5000,
  "currentPrice": 5000,
  "targetParticipants": 5,
  "targetPrice": 4500,
  "participantsToUnlock": 2,
  "priceUnlocked": false,
  "dealMessage": "2명만 더 모이면 4,500원"
}
```

프론트엔드는 최소한 `dealMessage`만 표시해도 1차 UX를 만들 수 있다.

## 5. 화면 적용 범위

새 화면을 만들 필요는 없다.

### 게시글 카드

기존 카드에 짧은 문구만 추가한다.

```text
2명만 더 모이면 4,500원
```

### 게시글 상세

참여 버튼 근처에 가격 달성 상태를 표시한다.

```text
현재 예상가 5,000원
5명 달성 시 4,500원
현재 3명 참여 중
```

### 게시글 등록

기존 등록 폼에 선택 항목을 작게 추가한다.

```text
거래 방식: 선모집형 / 후모집형
할인 조건 사용: ON/OFF
목표 인원
목표 달성 가격
```

## 6. 백엔드 변경 방향

기존 `groupBuyType`을 명확한 enum으로 좁힌다.

```ts
groupBuyType: "pre_recruit" | "post_recruit"
```

거래 세부 모드를 추가한다.

```ts
groupBuyMode: "normal" | "price_unlock"
```

가격 해금 조건을 저장한다.

```ts
targetParticipants: number | null
targetPrice: number | null
```

백엔드가 응답 계산 필드를 내려준다.

```ts
currentPrice: number
participantsToUnlock: number
priceUnlocked: boolean
dealMessage: string
```

## 7. 기존 A/B 구조와의 관계

기존 A/B 구조는 유지한다.

```text
A 타입: 선모집형
B 타입: 후모집형
```

이번 제안은 A 타입의 세부 기능으로 들어간다.

```text
A 타입: 선모집형
  - 기본 선모집
  - 모이면 싸지는 공구

B 타입: 후모집형
  - 기존 후모집 유지
```

후모집형의 다각화는 2차 확장으로 둔다.

```text
박스 쪼개기
잔여 수량 나눔
번개 수령
```

## 8. 기대 효과

사용자 입장에서는 참여 이유가 명확해진다.

```text
그냥 참여하는 공구
-> 내가 참여하면 가격 조건이 좋아지는 공구
```

서비스 관점에서는 공동구매의 색깔이 강해진다.

```text
단순 게시판형 공동구매
-> 참여자가 모일수록 조건이 좋아지는 공동구매
```

프론트엔드 관점에서는 부담이 작다.

```text
새로운 화면 구조 불필요
기존 카드/상세/등록 화면에 작은 필드만 추가
계산은 백엔드 담당
```

## 9. 후속 확장

1차는 한 단계 가격 해금만 제공한다.

추후 반응이 좋으면 다단계 가격표로 확장할 수 있다.

```text
3명 달성: 5,000원
5명 달성: 4,500원
8명 달성: 4,000원
```

후모집형은 별도 기능으로 확장한다.

```text
박스 쪼개기:
귤 10kg을 2kg씩 나눠 가짐

잔여 수량 나눔:
이미 구매한 상품 중 남은 수량만 모집

번개 수령:
오늘 특정 시간에 빠르게 모이고 바로 수령
```

## 10. 결론

거래 방식 다각화의 1차 기능으로는 **선모집형 + 모이면 싸지는 공구**가 가장 적합하다.

이유는 다음과 같다.

```text
공동구매의 본질과 잘 맞음
사용자 참여/공유 동기가 생김
프론트엔드 변경 부담이 작음
현재 백엔드 구조에 자연스럽게 붙일 수 있음
```

1차 목표는 다음 문구 하나를 서비스에 자연스럽게 넣는 것이다.

```text
2명만 더 모이면 가격이 내려가요.
```
